# N6 — 상용화 마스터플랜: 이커머스 캠페인 애널리틱스 툴 전면 개편

> 작성: 2026-06-10 · 기준 코드: main `c75b605` (N5.1 머지 직후)
> 목표: **타사에 판매해도 손색없는 수준**의 캠페인 플랜·실적 애널리틱스 제품으로 개편
> 근거: 코드베이스 전체 감사 + 상용 제품(Triple Whale, Polar Analytics, Northbeam, Airbridge 등) UX 리서치 + 2026 대시보드 설계 표준

## 0. 포지셔닝 — 왜 이 제품이 팔릴 수 있는가

시장 조사 결론: **"캠페인 플랜 vs 실적(예산 대비 달성)" 분석을 1급 개념으로 가진 제품이 국내외에 없다.**

- 미국 (Triple Whale $149~4,489/mo, Polar ~$300~500/mo, Northbeam $1,500+/mo): 광고 어트리뷰션·블렌디드 수익 중심. 플랜 대비 페이싱은 부속 기능.
- 한국 (Airbridge, Bigin, BizSpring): 앱 어트리뷰션·CRM 메시징 중심. 캠페인 목표 수립→실적 비교 워크플로 부재.
- 한국 마케팅 팀의 현실: 플랜은 여전히 엑셀. **엑셀 네이티브 임포트 + 플랜·실적 비교**가 우리의 웻지.

상용 제품이 "팔리는 느낌"을 주는 공통 요소 (이번 개편의 기준선):
1. 의견이 담긴 **프리빌트 리포트 팩** (빈 캔버스가 아니라)
2. **KPI 타일 = 수치 + 전기간 대비 델타 + 스파크라인 + 클릭 시 드릴다운** 패턴의 전면 적용
3. **제품이 먼저 말을 거는 기능**: 이상 감지·페이싱 알림·주간 다이제스트 (2026년 헤드라인 기능)
4. 연결 즉시 차오르는 대시보드 (time-to-value 수 분 이내)
5. 다크/라이트 + 밀도 토글, cmd-K 커맨드 팔레트 등 2026 SaaS 테이블 스테이크

## 1. 현재 상태 진단 (감사 결과 요약)

사용자 불만 6건 → 근본 원인:

| # | 불만 | 근본 원인 (감사로 확인) |
|---|------|--------------------------|
| 1 | 로딩이 매우 느림 (특히 히스토리 분석) | **머티리얼라이즈드 뷰/캐시 0개.** 모든 페이지가 매 요청마다 24개 캠페인의 업리프트·헤일로·기여도를 SQL로 풀 재계산. `force-dynamic` + `router.refresh()` 17곳 → 저장할 때마다 전체 재계산. 캠페인 상세는 13개 쿼리를 4블록 순차 실행 + 드롭다운 하나에 전 캠페인 스캔(`campaigns_with_actuals` 무필터). 데이터는 작음(daily_sales 27K행) — 구조 문제라 완전 해결 가능 |
| 2 | 다이어그램(차트) 부족 | 차트가 대시보드 홈에만 집중. /predict·/prescribe = 차트 0개, /library = 조건부 미니차트 1개. 상세 페이지도 표 위주 |
| 3 | 캠페인 연동 뎁스 과다 | 플랜↔실적 연동·SKU 매칭·병합이 3개 화면(상세/플랜에디터/편집)에 분산. SKU 수동 매칭은 5+클릭·2페이지 왕복 |
| 4 | 시트명이 워싱됨 | 원본 파일명은 DB(`upload_log.source_file`)에 **그대로 저장돼 있음**. 화면 곳곳에서 파일명 대신 시트 내용에서 추출한 캠페인 코드/라벨을 표시하는 UI 문제. 업로드 이력 ↔ 생성된 데이터 간 연결 표시도 없음 |
| 5 | 비주얼 무변화 | N5는 데이터 모델 작업이었음(정상). 단, 디자인 시스템 부재가 구조적 문제: 공용 components/ 없음, 페이지 폴더에 컴포넌트 산재, 디자인 토큰과 Tailwind 유틸 혼용 |
| 6 | 자잘한 사용성 | 에러 = 전부 `alert()` (7곳), 낙관적 업데이트 0, 페이지네이션·가상화 0, ARIA·키보드 내비 0, 빈 상태 일부 누락 |

**현 스택**: Next.js App Router + Tailwind v4 + Recharts 3.8 + Supabase (마이그레이션 0001~0021). react-query/SWR 없음.

## 2. 설계 원칙 (전 페이즈 공통)

1. **3-레이어 IA** (Shneiderman: overview → zoom/filter → details-on-demand)
   - L1 홈: 5~7개 핵심 KPI만. "지금 괜찮은가?"에 3초 내 답
   - L2 진단: 캠페인 리스트 — 플랜 이탈도(페이싱) 기준 상태 색상
   - L3 상세: brush/zoom 시계열 + 원본 행 데이터까지 드릴
2. **모든 숫자는 맥락과 함께**: 단독 수치 금지. 델타(전기간/플랜 대비) + 스파크라인 + 클릭 드릴
3. **표가 곧 제품**: 엑셀 기반 데이터 특성상 데이터 그리드에 차트만큼 투자 (셀 내 달성률 바, 컬럼 피닝, 스티키 헤더)
4. **선조작 후보고**: 모든 변이는 낙관적 업데이트 + 토스트. `alert()`/전체 refresh 금지
5. **요청 시점 집계 금지**: 업로드(인제스트) 시점에 롤업 계산 → 대시보드는 서빙 테이블만 읽음
6. **색은 상태에만**: 수치 인코딩은 위치/길이로, 색상은 좋음/주의/나쁨 시맨틱 전용 (NN/g)

## 3. 페이즈 계획

### R1 — 체감 수술 (즉효: 속도·뎁스·파일명·토스트)

> 목표: "느리고 불편하다"를 1~2 세션 안에 제거. 비주얼 개편 전이라도 매일 쓰고 싶은 도구로.

**R1.1 성능: 롤업 서빙 테이블 도입** ⚡ 최우선
- 마이그레이션 0022: `campaign_rollups` (캠페인별 summary/achievement/fit 사전 계산), `daily_rollups` (일자별 집계)
- 갱신 트리거: 업로드 완료 시 + 플랜 확정/연동 변경 시 해당 캠페인만 재계산 (데이터가 배치 유입이므로 pg_cron보다 인제스트 시점 계산이 정확·저렴. pg_cron은 야간 정합성 보정용 1회만)
- 페이지당 RPC 1개로 통합: `dashboard_bundle()`, `library_bundle()`, `promotion_detail_bundle(p_id)` — JSON 번들 반환
- 캠페인 상세 13쿼리 → 1~2 RPC. `campaigns_with_actuals` 무필터 스캔 제거
- 수용 기준: **모든 페이지 서버 응답 < 300ms, 체감 로드 < 1초**

**R1.2 RSC 스트리밍 구조**
- 위젯 단위 `<Suspense>` 분리: 느린 위젯 하나가 페이지 전체를 막지 않게
- `router.refresh()` 17곳 → 변이 후 해당 데이터만 무효화 (TanStack Query 도입, 서버 prefetch + HydrationBoundary)
- 필터 상태는 URL searchParams로 (공유 가능한 뷰)

**R1.3 업로드 이력·파일명 투명화**
- 업로드 이력에 원본 파일명을 1급 컬럼으로: 파일명 + 종류 + 행수 + **생성/갱신된 캠페인 링크** + 시각
- 캠페인 상세에 "이 캠페인의 데이터 출처" 섹션: 어떤 파일이 언제 무엇을 만들었는지 역추적
- `daily_sales.source_file` ↔ `upload_log` 연결 강화 (upload_log_id FK)

**R1.4 연동 뎁스 압축: "연동 센터"**
- 캠페인 상세 안에 플랜 연동·SKU 매칭·병합을 **단일 패널**로 통합 (현재 3화면 분산)
- SKU 매칭: 좌(플랜 SKU)·우(실적 SKU) 2컬럼 + 정규화 자동 매칭 제안 미리보기 + 원클릭 수락. 5+클릭 → 2클릭
- 미연동 항목은 목록(L2)에서 배지로 노출 → 클릭 시 바로 연동 패널 딥링크

**R1.5 피드백 시스템**
- `alert()` 7곳 전부 → 토스트(sonner) + 실패 시 재시도 버튼
- 변이는 낙관적 업데이트, 파괴적 행동(삭제/병합/확정)만 확인 다이얼로그 유지

### R2 — 디자인 시스템 + IA 전면 개편 (비주얼 대수술)

> 목표: 첫 화면에서 "상용 제품"으로 보이게. 모든 메뉴에 시각화.

**R2.1 디자인 시스템: shadcn/ui 전환**
- shadcn/ui + Tailwind 토큰 체계로 전면 전환. 코랄 브랜드 컬러는 유지하되 **뉴모피즘 → 밀도 높은 모던 플랫**으로 전환 (상용 애널리틱스 표준: 중립 표면 + 포인트 컬러. Tremor 블록 패턴 차용 — Vercel 인수 후 MIT 전환, 복사해 쓰는 패턴 라이브러리로 활용)
- 공용 `components/` 디렉토리 신설: KpiCard, DataGrid, ChartCard, StatusBadge, EmptyState, PageHeader 등
- 다크 모드 + 밀도 토글(comfortable/compact), `tabular-nums` + `Intl.NumberFormat('ko-KR')` 억/만 단위 포맷터 통일
- 접근성 기본기: ARIA, 키보드 내비, 색상 외 상태 표시(아이콘+텍스트)

**R2.2 IA 재편 (3-레이어)**
- **홈(L1)**: KPI 타일 5~7개 (기간 매출, 플랜 달성률, 업리프트, 진행 중 캠페인 페이싱, 기여이익) — 각 타일 = 수치+델타+스파크라인+클릭 드릴. 좌상단에 가장 중요한 지표(F-패턴). 하단에 "주의 필요" 캠페인 모듈
- **캠페인(L2)**: 리스트를 TanStack Table 기반 DataGrid로 — 셀 내 달성률 바, 페이싱 상태 색상, 컬럼 피닝/정렬/저장되는 뷰. 라이브러리(히스토리 분석)와 캠페인 목록의 역할 중복 정리
- **캠페인 상세(L3)**: 상단 KPI 스트립 → ECharts 메인 시계열(brush/zoom, 크로스헤어, 플랜 라인 vs 실적 라인 오버레이) → 탭 구조(실적/플랜 비교/연동 센터/메모·출처)
- **cmd-K 커맨드 팔레트**: 캠페인 점프, 기간 전환, 페이지 이동, 업로드 바로가기

**R2.3 차트 2-티어 전략**
- 경량: shadcn charts(Recharts 래퍼) — KPI 스파크라인, 작은 바/도넛 (테마 자동 통합)
- 고급: **ECharts (tree-shaken ~100KB gz, SVG 렌더러)** — 캠페인 상세 시계열, 히스토리 비교 등 brush/zoom·드릴이 필요한 2~3 화면만
- 시각화 공백 메우기:
  - /predict 시뮬레이터: 예측 결과 분포 차트 + 민감도(입력 변화→결과) 시각화
  - /prescribe 추천: 추천 근거 점수 바차트 + 유사 과거 캠페인 산점도
  - /library: 캠페인 비교 오버레이 차트(복수 선택 → 기간 정규화 비교), 목적별 성과 히트맵
  - 상세: 플랜 vs 실적 워터폴/누적 페이싱 차트

### R3 — 애널리틱스 깊이 (제품이 먼저 말 걸기)

> 목표: "보러 가는 대시보드"에서 "알려주는 제품"으로. 2026 상용 제품의 핵심 차별 요소.

**R3.1 페이싱 알림 + 주간 다이제스트** (Moby AI에 대한 저비용 대항마 — 룰 기반으로 충분)
- 진행 중 캠페인: 경과일 대비 목표 진척률 계산 → "캠페인 X: 기간 60% 경과, 목표 매출 42% — 주의" 류의 룰 기반 알림
- 종료 캠페인: 자동 회고 카드 (달성률, 전 캠페인 대비, 특이점)
- 주간 한국어 다이제스트: 이메일(Resend) 또는 슬랙 — 대시보드가 사용자를 찾아가게
- 홈 상단 "인사이트 피드"에 동일 내용 노출

**R3.2 엑셀 임포트 위저드** (OneSchema/Flatfile 패턴 — 활성화의 순간)
- 업로드 → **컬럼 매핑 단계** (한국어 헤더 동의어 자동 인식: 집행액/소진액/광고비 등, 지난 매핑 기억·재사용) → 검증 프리뷰(에러 행 인라인 수정) → 임포트 요약
- 현재 5종 시트 포맷 고정 의존 → 포맷 변형에 강건해짐 (판매 시 필수: 고객마다 시트가 다름)

**R3.3 온보딩 + 내보내기**
- 빈 상태 → 데모 데이터 1클릭 생성 + 3단계 체크리스트 (시트 업로드 → 매핑 확인 → 대시보드 보기)
- 모든 그리드/리포트 엑셀 내보내기 (한국 구매자는 엑셀-아웃을 반드시 요구)

### R4 — 상용화 (멀티테넌트 SaaS)

> 목표: 실제 판매 가능 상태. 페이잉 고객 신호가 있을 때 착수.

- **멀티테넌시**: 전 테이블 `org_id` + RLS (`org_id = auth.jwt()->>'org_id'`), 정책 컬럼 인덱스, 역할 4종(Owner/Admin/Member/Viewer). 에이전시용 멀티 브랜드 워크스페이스 (Polar의 global views 패턴)
- **감사 로그**: append-only `audit_log` (행위자/조직/액션/before-after JSON) — 업로드 분쟁 디버깅 + 기업 요구사항
- **공유 리포트 링크**: 브랜딩(로고+컬러) 적용된 읽기 전용 캠페인 리포트 URL — 에이전시가 돈 내는 기능
- **i18n**: next-intl, ko 기본 + en. 숫자/통화 포맷은 R2에서 이미 분리
- **빌링**: 국내 Toss Payments/PortOne + 글로벌 Stripe. 과금 축은 관리 광고비 또는 캠페인 수 (카테고리 표준 = 가치 기반)
- **SSO**: Kakao/Naver/Google
- 보류 (고객이 요구할 때까지): 커스텀 도메인 화이트라벨, 읽기 전용 복제본

# UX 편의성 기준 (캠페인 애널리틱스 — 항상 최소 이 수준으로 개발)

MD가 실제로 쓰면서 기대하는 편의성. 새 기능/수정 시 기본값으로 지켜라.

- **입력값 보존**: 사용자가 넣은 숫자는 그대로 유지한다. 내부 계약(예: set_price=Σ(sku단가×수량)) 때문에 표시값이 임의로 바뀌면 안 됨(예: 89,900 입력 → 89,898로 자동 변형 금지). 필요하면 내부 단가를 소수까지 허용해 합계를 입력값과 정확히 일치시킨다.
- **할인율·비율 표기**: 소수점 없이 정수로, 내림(floor) 표기. (`pctFloor` 사용)
- **금액 표기**: 천 단위 콤마. 입력 필드도 콤마 표시 권장(`won`/`num`/`commaFmt`).
- **저장 상태 가시화**: 저장 완료면 입력 비활성 + 버튼을 '수정'으로, '수정' 누르면 다시 편집 + '저장'. 업로드류는 파일명·시각을 보여준다.
- **차감/반영액 명시**: 매출·공헌이익 등 핵심 수치 아래에 쿠폰 할인 반영액·사은품 차감액 등 조정 내역을 함께 표기.
- **조건부 표시 배지**: 무료배송(무배)·쿠폰 적용(쿠폰1/2) 등 자동 적용 조건은 배지로 즉시 보이게.
- **목록 조작**: 항목은 자유롭게 추가/삭제, 드래그로 순서 변경, 의미 있는 자동 정렬(메인↑/서브↓ 등)을 제공.
- **확정 등 되돌리기 어려워 보이는 액션**: 확정 전 확인 팝업 + "확정 후에도 변경 가능" 안내, 완료 후 자연스러운 다음 화면으로 이동.
- **버튼·라벨은 사용자의 멘탈모델대로**: 전문용어(예: '어태치율') 대신 쉬운 말('함께 담은 비율'). '편집'보다 '보기' 후 그 안에서 수정하는 흐름이면 '보기'로.
- **알림/버튼은 반드시 동작**: 클릭해도 아무 일 없는 CTA 금지(해당 패널로 스크롤/이동·자동 펼침까지 연결).
- **계산 단일 출처**: 금액·공헌 계산식은 `lib/plan.ts`에 두고, API 라우트와 `confirm_plan`(SQL)이 동일 식을 쓰게 유지. 새 DB 컬럼은 best-effort(미적용 환경에서도 안 깨지게).

## 3. ★ 현재 막힌 지점 — 플랜/실적 강결합

### 3.1 사용자가 발견한 진짜 문제

> "플랜과 실적 시트가 서로 다른 코드로 업로드돼 별개의 `promotions` 행이 생긴다.
> 머지하려고 하면 양쪽 모두 플랜이 있어서 실패.
> 알고 보니 캠페인 상세에서 **'플랜 편집' 버튼을 누른 것만으로 자동 draft 플랜이 생성**돼서 그렇다.
> 플랜을 수동으로 입력하는 건 얘기한 적 없는 기능이다."

### 3.2 강결합의 구체 사례 (운영 DB 기준)

- **CF_P_260518_벌크UP** (2026-05-18 ~ 05-19): 실적 48 SKU 있음. 플랜 없는 상태에서 사용자가 상세 → "플랜 만들기" 클릭 → 빈 draft 플랜 생성됨. (`app/api/promotions/[id]/plan/route.ts` POST)
- **CF_P_260511_벌크UP** (2026-05-10): draft 플랜 10 SKU 있음 (가이드 ⑤ 임포터로 적재됨). 실적 없음.
- 사용자: 두 캠페인이 같은 행사라고 보고 머지 시도 → 양쪽 모두 플랜이라 0019 함수가 거부.

### 3.3 코드 흐름의 근본 원인

1. `app/(dash)/promotions/[id]/page.tsx` 의 "플랜 만들기" 링크 → `/promotions/[id]/plan`
2. `app/(dash)/promotions/[id]/plan/page.tsx` 로드 → `app/api/promotions/[id]/plan/route.ts POST` 호출
3. POST 가 "플랜 없으면 version=1 draft(current) 생성" → **버튼 클릭만으로 빈 플랜 row 생성**
4. 이후 머지/비교 로직이 "플랜이 있다" 로 판정 → 강결합

### 3.4 사용자 요구 새 구조

> "플랜과 실적에 대한 시트 데이터 자체를 별도로 운영해야 할 것 같아."

해석: **플랜과 실적은 서로 독립적인 업로드 아티팩트**. 같은 `promotion_id` 에 강결합하지 말고:

- **플랜은 플랜대로** 적층 (자체 식별자: code/name/기간)
- **실적은 실적대로** 적층 (자체 식별자: code/name/기간)
- **비교**는 사용자가 명시적으로 1:1 짝지어 같은 형식으로 봄

지금 시도한 N4.3 (`campaign_plans.actual_promotion_id`) 은 같은 방향이지만 **여전히 plan→promotion 의존을 떠나지 못함**. 사용자가 원하는 건 더 근본적인 분리.

## 4. 다음 작업 — 사용자 지시서 대기 중

사용자가 새 채팅에서 새 개발 지시서를 작성해 올 예정. 다음 시나리오 중 하나로 보임:

- `campaign_plans.promotion_id` → nullable 또는 제거

- 비교는 두 캠페인을 쌍으로 묶음

### 시나리오 C: 더 가벼운 수정
- 자동 draft 플랜 생성 제거 (POST `/api/promotions/[id]/plan` 동작 변경)

4. 운영 DB의 빈 draft 플랜들 청소 SQL 미리 준비 (다음 작업 전 사전 정리)

```sql
-- 빈 draft 플랜 확인 (옵션 0개)
select cp.id, cp.promotion_id, cp.status, p.name
from promo.campaign_plans cp
join promo.promotions p on p.id = cp.promotion_id
left join promo.campaign_plan_options cpo on cpo.campaign_plan_id = cp.id
where cp.status = 'draft'
group by cp.id, cp.promotion_id, cp.status, p.name
having count(cpo.id) = 0;
```

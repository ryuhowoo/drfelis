## 0. 한 줄 요약

캠페인 **사전 계획(가격 가이드)** 과 **사후 실적** 의 갭(= **달성률**)을 추적하고, 모든 분석을 **목적(purpose)** 으로 슬라이스하는 기능을 promo-analytics에 추가한다. 기존 측정 v2 엔진·페이지 위에 **2계층(가격 마스터 ↔ 캠페인 플랜)** 을 얹는다.

## 1. 이번 작업의 범위

| 포함 (v1)                            | 제외 (다음 단계)                    |
| ---------------------------------- | ----------------------------- |
| 가격 마스터 적재(품목 + 단품/N묶음/무배 구성)       | 정기구독 구성 적재·플랜                  |
| 캠페인 플랜(옵션=다중 SKU BOM) + 예상 매출/공헌이익 | 추가구성(크로스셀 5%) 플랜              |
| 달성률(SKU 단위 코어 / 옵션 단위 보조)          | 달성률 패턴을 예측 엔진 신뢰도에 가중(S6, 장기) |
| 대시보드·히스토리 달성률 노출                   | 작년 동기간 시즌 보정 (데이터 미보유)        |
| 목적 슬라이스 전반화                        |                               |

## 2. 확정 설계 결정 (의사결정 로그 — 변경 시 여기부터 고친다)

1. **공헌이익 공식 (자사몰 고정 rate card)**
   - 변동비율(매출 대비) = 수수료 **4.5%** + 광고비 **10%** + 물류비 **12%** + 적립금 **2%** = **28.5%**
   - 원가 = **원가(VAT+)** (절대액), 그 외는 매출 비례.
   - **공헌이익 = 매출 × 0.715 − 원가합**, **공헌이익률 = 공헌이익 / 매출**
   - rate card는 코드 상수가 아니라 **DB 파라미터**(`promo.rate_card`)로 둔다. 캠페인 플랜 확정 시 그 시점 rate card를 **스냅샷**해 과거 달성률이 흔들리지 않게 한다.
   - ⚠️ 원본 워크북의 `공헌이익` 시트는 10/10/15(=35%)로 계산돼 있으나 **구버전/외부채널로 간주하고 무시**한다.
2. **할인율 이중 표기**: 모든 구성에서 **소비자가 기준 할인율**(주 표기) + **상시가 기준 할인율**(내부 확인용) 둘 다 노출.
3. **2계층 데이터 모델**
   - **L1 가격 마스터**: 워크북에서 주기적 재업로드. `products`(품목 시트) + `product_price_configs`(SKU × 구성).
   - **L2 캠페인 플랜**: `campaign_plans`(헤더, draft↔confirmed) + `campaign_plan_options`(옵션) + `campaign_plan_option_items`(옵션 BOM = 다중 SKU + SKU별 수량·단가).
4. **옵션 = BOM**: 한 옵션은 여러 SKU의 묶음. 컴포넌트마다 `sku_qty_per_option`(세트당 SKU 수량)과 `unit_sale_price`(옵션 내 SKU 단가)를 기입. 옵션 세트 단가 = Σ(수량×단가). **옵션별 판매수와 SKU별 판매수를 둘 다** 표시.
5. **플랜 라이프사이클**: `draft`(자유 수정, 대화하며 다듬음) → `confirmed`(불변 스냅샷 + rate card 스냅샷). 확정 후 수정은 새 버전 생성.
6. **달성률 매칭 규칙**
   - **코어 = SKU 단위**: `promotion_sales`를 `product_id`로 합산 → 플랜의 SKU 예상치와 비교. (가장 신뢰 가능한 조인키)
   - **보조 = 옵션 단위**: `promotion_sales.option_info` ↔ 플랜 옵션 라벨 매칭. 자동 매칭 실패 시 "옵션 매핑 필요" 표시 + 수동 매핑 제공. v1은 SKU 단위를 1차 지표로.
   - 달성률 3종: 매출 / 수량 / 공헌이익. `actual / expected`, 분모 0·미판매·계획외 항목 처리 규칙은 S3에서 명시.
7. **목적 가중(목적 슬라이스)**: `promotions.purposes[]` 사용. **목적 가중 방식** 채택 — 각 캠페인에 목적별 가중치(주목적 1.0, 보조목적은 가중)를 부여하고 집계·랭킹·예측을 가중 적용. 보조목적 기본 가중치(0/0.5 등 정확값)는 **S5 착수 시 1줄 확정**. (스키마는 가중치를 담을 수 있게 미리 설계)
8. **데이터 부족 배지**: 수량 의존 지표(재고소진·브랜딩 신규비중 등)는 현재 `daily_sales.quantity=0`·`option_info=""` 때문에 신뢰도 낮음. 해당 지표엔 **"데이터 부족" 배지**를 달고 계산은 시도하되 경고. (사용자가 수량 데이터 보충 예정)
9. **재업로드 중복방지**: 가격 마스터는 **upsert**(고유키 충돌 시 갱신). 다른 `source_file` 데이터 존재 시 PR #27의 confirm 패턴 재사용.
10. **용어**: 화면 텍스트는 **"캠페인"** 통일. URL(`/promotions/`)·DB 스키마(`promotions`/`promotion_*`)·변수명은 호환 유지(HANDOFF 결정 계승).

## 3. 데이터 모델 전체도 (개요 — 상세 DDL은 각 세션)

```
[기존]
products ───┐
            ├─< daily_sales            (baseline 연료)
promotions ─┼─< promotion_main_products
            ├─< promotion_sales        (사후 실적, 달성률의 actual)
            └─< promotion_notes        (집요하게 묻기)

[신규 — L1 가격 마스터]
products ─< product_price_configs       (SKU × 구성: 가격·할인율·단위공헌이익)
promo.rate_card                          (변동비율 파라미터, 단일 행 + 이력)

[신규 — L2 캠페인 플랜]
promotions ─1:1─ campaign_plans                 (헤더: status, rate_card 스냅샷, 합계)
campaign_plans ─< campaign_plan_options          (옵션: 라벨, 예상 세트수)
campaign_plan_options ─< campaign_plan_option_items   (BOM: product_id, 세트당수량, 단가)
```

세션별 신규 객체:

- **S1**: `product_price_configs`, `promo.rate_card`, `products` 정합(원가 VAT+)
- **S2**: `campaign_plans`, `campaign_plan_options`, `campaign_plan_option_items` + 예상치 계산 함수
- **S3**: `promo.plan_vs_actual(p_id)` 함수(달성률)
- **S5**: 목적 가중 컬럼/뷰

## 4. 공통 컨벤션 · 제약 (모든 세션 강제)

1. **AGENTS.md 최우선** — 코드 작성 전 반드시 `node_modules/next/dist/docs/`의 관련 가이드를 읽고 시작. "This is NOT the Next.js you know." (Next 16.2.7 / Turbopack / App Router)
2. **디렉토리**: 모든 코드는 `promo-analytics/` 아래. 스키마는 `promo`.
3. **마이그레이션**: additive only. 다음 번호 **0006** 부터. **Supabase MCP `apply_migration`로 클로드코드가 직접 적용**(SQL Editor 수동 X). 파괴적 변경 금지(drop/rename 지양, 컬럼 추가 위주).
4. **RLS**: 신규 테이블 모두 `authenticated` 역할에 `for all using (true) with check (true)` (기존 패턴 동일).
5. **타입**: 신규 테이블·행 타입은 `lib/types.ts`에 반영. 파싱은 `lib/parse.ts` 확장(헤더 자동 인식 + dry-run 미리보기).
6. **스택 유지**: recharts / xlsx(브라우저 파싱) / `@fontsource-variable/asta-sans` 폰트체인 그대로. 새 무거운 의존성 추가 금지(추가 필요 시 지시서에서 사전 합의).
7. **머지 워크플로**: branch → push → PR → **Vercel 프리뷰 빌드 Ready 확인** → 머지. 1 세션 = 1 PR.
8. **세션 종료 규약**: 각 세션 마지막은 항상 **"이 섹션 완료. 검수 후 다음 지시 대기."** 로 끝낸다. 한 세션에서 다음 세션 범위로 넘어가지 않는다.

| 세션     | 범위                         | 완료 게이트(검수 기준)                                                                               |
| ------ | -------------------------- | ------------------------------------------------------------------------------------------- |
| **S1** | 가격 마스터 스키마 + 업로드/파서        | 워크북 업로드 → `products` 정합 + `product_price_configs` 적재. dry-run 미리보기 통과. 재업로드 시 중복 없이 upsert. |

| **S5** | 목적 슬라이스 전반화                | 대시보드·히스토리·시뮬레이터·상세·추천을 목적으로 슬라이스(가중 적용). 수량 의존 지표 "데이터 부족" 배지.                              |

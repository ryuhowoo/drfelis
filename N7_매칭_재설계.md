## 0. 왜 다시 짜는가 (현재 실패의 실증)

운영 데이터(Better Habits 플랜 `7e96ef4f`, 실적 `626e9550`)에서 확인된 실제 문제:

### 플랜 쪽 (가이드 ⑤가 만든 옵션)
같은 라벨인데 서로 다른 묶음이 별개 옵션으로 존재 — **라벨만으론 구분 불가**:
| option_label | set_price | composition(저장값) |
|---|---|---|
| (제품) 데일리 솔루션 퍼펙트 치킨 150g [15p] | 82,708 | 퍼펙트 치킨 ×6 |
| (제품) 데일리 솔루션 퍼펙트 치킨 150g [15p] | 33,000 | 퍼펙트 치킨 ×2 |
| (제품) 데일리 솔루션 퍼펙트 치킨 150g [15p] | 82,702 | 퍼펙트 치킨 ×6 |

구성 오염도 실재:
- "퍼펙트 **연어**" 옵션의 구성이 "퍼펙트 **치킨** ×6"으로 잘못 저장
- "7+ 케어 **치킨**" 옵션 구성이 "퍼펙트 치킨 ×2"로 엉뚱하게 매핑
- 이름 불일치: `(제품) 데일리 솔루션 퍼펙트 연어` / `데일리솔루션 퍼펙트 연어` / `세븐플러스 7+ 케어` — 같은 상품, 다른 문자열

### 실적 쪽 (③ 매출 시트 option_info — 쇼핑몰 자유 텍스트)
```
상품선택=퍼펙트 케어 치킨, 수량=6박스 (3개월)
상품선택=세븐플러스 케어 연어, 수량=2박스 (1개월)
상품선택=마스터, 중량/수량=8.3kg/4개
상품선택=❤️미야옹철 추천❤️[38%⬇️] 마스터 8.3kg 2개
상품선택1=퍼펙트, 상품선택2=치킨 2박스 (1개월분)
```
→ 상품 + 묶음수(2박스/4개/6박스) + 할인문구 + 이모지가 뒤섞임. base_name으로 SKU는 알 수 있으나 **묶음수는 자유 텍스트에 파묻혀 있음**.

### 근본 원인 3가지
1. **옵션 정체성을 "대표 SKU 라벨"로 잡음.** 옵션 = (상품 × 묶음수 × 단품/혼합)인데 라벨 하나로 식별 → 2/6개입 충돌, 혼합 표현 불가.
2. **임포터가 1행=1옵션=단일 SKU 모델.** 혼합 옵션을 단일 product에 욱여넣어 구성 오염. item_code 누락/이름 충돌 시 오매핑.
3. **매칭을 "라벨 문자열 부분일치"로 시도.** 양쪽 다 묶음수가 구조화돼 있지 않아 비교 불능 → 옵션 단위 달성률 전부 0%.

---

## 1. 설계 원칙

1. **달성도의 1차 진실은 SKU(품목) 단위.** product_id(+정규화 이름)로 안정적으로 계산. "퍼펙트 치킨: 예상 N박스 vs 실제 M박스, 달성률".
2. **묶음(pack)은 파생 차원으로 구조화.** 플랜은 옵션 구성에서, 실적은 option_info 파싱으로 묶음수를 뽑아 (상품×묶음) 단위 비교를 *부가* 제공. 못 뽑으면 SKU 단위로 안전 폴백.
3. **옵션은 구성으로 식별·표시.** 라벨이 아니라 components[(상품, 세트당수량)] 시그니처로 동일성 판단, 화면엔 항상 "구성 + 묶음 + 세트가"를 노출.
4. **매칭은 제안 + 확정.** 자동 추천(상품·묶음 일치도 순) 후 사용자가 한 번에 확정/보정. 빈손에서 드롭다운 헤매기 금지.
5. **모든 미매칭은 명시.** 양쪽 미매칭 SKU·옵션을 버킷으로 분명히 보여주고, 달성률에 "미매칭 제외" 영향 표기.

---

## 2. 데이터 모델 (목표)

### 2.1 플랜 옵션 = 구성 기반 번들
`campaign_plan_options` (유지) + `campaign_plan_option_items` (핵심화):
- 옵션은 **components 1..N**: `(product_id, units_per_set)` — 혼합(치킨3+연어3) 네이티브 표현
- `set_count` = 예상 세트 수 (기존 expected_option_qty)
- `set_price`, 경제값(기존 econ) 유지
- **option_signature** (신규, 파생): `sorted(product_id:units_per_set)` 해시 → 2개입/6개입/혼합이 각각 고유. 동일 시그니처 중복 옵션은 임포트시 합산/경고.
- **display_label** (신규, 파생): 구성 기반. 단품 "퍼펙트 치킨 6박스", 혼합 "치킨3+연어3(혼합)". 원본 라벨은 raw_label로 보존(참고용).

### 2.2 실적 묶음 파싱
신규 `promo.parse_pack(option_info text) → int` (SQL) 또는 적재시 컬럼화:
- "(\d+)\s*(박스|개|입|팩)" 추출, "8.3kg/4개"의 4, "6박스 (3개월)"의 6 등
- `promotion_sales`에 `pack_size int` 파생 컬럼 추가(적재시 채움, 과거분 백필). 못 뽑으면 null.
- product_id는 기존대로 base_name/정규화로 해결(이미 동작).

### 2.3 매칭 테이블
- SKU 단위 수동 매핑: 기존 `promotion_sku_mappings` 유지 (product_id ↔ product_id).
- 옵션 단위 매핑(신규, 선택): `promotion_option_mappings(plan_option_id, actual_signature)` — (상품집합+pack) 키로 확정. 자동 추천이 빗나간 것만 저장.

## 3. 매칭 & 달성 계산 (목표)

### 3.1 SKU(품목) 달성 — 1차·신뢰
- 플랜 기대수량(상품별) = Σ_options(set_count × units_per_set) over 그 상품이 든 옵션
- 실적 수량(상품별) = Σ promotion_sales.quantity (매핑 redirect 반영)
- 달성률 = 실적/기대. 매출/공헌도 동일.
- 정규화 이름·수동 매핑으로 product_id 차이 흡수 (기존 normalize_sku_name 재사용).

### 3.2 옵션(묶음) 달성 — 부가·best-effort
- 플랜 옵션 시그니처 = {(상품, units_per_set)}
- 실적을 (product_id, pack_size)로 그룹 → 시그니처 후보 구성
- 매칭: 단품 옵션은 (상품, pack=units_per_set) 정확 일치; 혼합은 구성 상품 동시 등장 + pack 일치 추정
- 못 맞춘 실적/옵션은 미매칭 버킷. 옵션 달성률은 "참고"로 표기, 달성도 총평은 3.1 기준.

---

## 4. UI/UX 재설계

### 4.1 옵션 표시 (지적사항 직접 해결)
모든 옵션 행에 **구성·묶음·세트가**를 노출:
```
퍼펙트 치킨  ·  6박스/세트  ·  82,708원   예상 50세트 → 실적 48세트 (96%)
치킨3+연어3(혼합)  ·  81,900원           예상 35세트 → 실적 — (미매칭)
```
라벨 단독 표기 금지. 2개입/6개입/혼합이 한눈에.

### 4.2 매칭 패널
- 좌: 플랜 옵션(구성·묶음·예상). 우: 추천 실적 그룹(상품·pack·실매출), 일치도순, ★ 고신뢰 프리셀렉트.
- 행 내 즉시 확정(현 인라인 방식 계승) + 미매칭 버킷 분리.
- SKU 탭 / 옵션 탭 분리: SKU 탭이 기본(신뢰), 옵션 탭은 묶음 분해 보기.

### 4.3 캠페인 상세 달성 블록
- 상단: 캠페인 총 매출/공헌 달성률 (SKU 기준).
- 표: 상품별 예상 vs 실적 vs 달성률 (+ 묶음 분해 펼치기).

## 5. 가이드 양식(⑤) 변경

현 양식은 1행=1옵션=단일 SKU라 혼합·묶음 표현이 빈약. 두 방향 중 택1:
- **(A) 양식 보강**: 옵션 행에 `구성(품목코드:수량, …)` 컬럼 추가 → 혼합·묶음 명시. 임포터가 이를 components로 파싱. (권고 — 양식 1컬럼 추가로 정확도 급상승)
- **(B) 현 양식 유지 + 추론 강화**: 혼합 행(품목코드 공란, "A+B" 라벨)을 같은 캠페인의 단품 행들로 역추론. 깨지기 쉬움.

→ **(A) 권고.** 표준 템플릿에 구성 컬럼 추가 + 다운로드 템플릿 갱신.
→ 단, 막힘 방지: **임포터는 두 양식 모두 지원**(구성 컬럼 있으면 정확 파싱, 없으면 추론 폴백). 사용자가 시트를 당장 안 바꿔도 동작.

## 6. 단계별 실행

- **P1 — 모델/적재 기반**: option_items 다구성 지원 + option_signature/display_label 파생, promotion_sales.pack_size 파생+백필, parse_pack(). 임포터(A안) 구성 컬럼 파싱 + (없으면) 추론. (마이그레이션 + parsePlanGuide + upload 임포터)
- **P2 — 달성 계산**: SKU 1차 달성 RPC 재작성(신뢰), 옵션 부가 달성 RPC. 롤업/번들 반영 + refresh_rollups(true) 검증.
- **P3 — UI**: 옵션 구성·묶음 표시, 매칭 패널 재구성(SKU/옵션 탭), 상세 달성 블록.
- **P4 — 데이터 정리**: 기존 오염 옵션(연어→치킨 등) 재적재 또는 보정 스크립트. 기존 campaign_plan 재임포트 안내.

각 단계 독립 PR. P1·P2는 하위호환(기존 함수 시그니처 유지하며 내부 교체) 우선.

## 7. 하위호환 / 데이터 보존
- promotions/promotion_sales/daily_sales/products/측정엔진/롤업: 변경 없음(컬럼 추가만).
- 기존 campaign_plan_options: option_items 1건짜리는 그대로 단품 옵션으로 동작. 혼합/오염분만 재적재 권장.
- 마이그레이션은 add column + 신규 함수 위주, 파괴적 변경 없음.

## 8. 확정한 설계 결정 (이의 없으면 이대로 구현)
1. **SKU 단위가 달성도 1차 진실, 옵션(묶음)은 부가.** (자유텍스트 실적에서 묶음 완전매칭은 불가 → 신뢰 가능한 SKU를 기준선으로)
2. **옵션 식별 = 구성 시그니처, 표시 = 구성+묶음+세트가.** 라벨 단독 금지.
3. **실적 묶음수는 option_info 파싱으로 구조화**(pack_size 파생 컬럼).
4. **가이드 양식에 '구성' 컬럼 추가(A안)** — 단, 임포터는 미존재시 추론 폴백.
5. **앱/엔진/데이터 보존**, 매칭·플랜옵션 레이어만 교체.
